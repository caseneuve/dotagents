## Context

`ChannelBackend` combines two responsibilities: messaging (publish/read/ack) and
display (status/progress/log/notify). Network transport needs these split —
messaging goes remote, display stays local.

## Acceptance Criteria
- [ ] `MessageTransport` interface: `publish`, `read`, `ack`, `subscribe`, `unsubscribe`
- [ ] `StatusDisplay` interface: `setStatus`, `setProgress`, `clearProgress`, `log`, `notify`
- [ ] `FileTransport` implements `MessageTransport` (extracted from current backends)
- [ ] `CmuxDisplay`, `TmuxDisplay`, `NoopDisplay` implement `StatusDisplay`
- [ ] `index.ts` wired with separate `transport` + `display` objects
- [ ] `ChannelPoller` becomes `FileTransport` implementation detail
- [ ] `FileTransport.subscribe()` accepts configurable interval (preserves `interval_seconds` from `channel_watch`)
- [ ] All existing tests pass, no behavior changes

## Affected Files
- `pi/extensions/agent-channel/backends.ts` → split into `transports.ts` + `displays.ts`
- `pi/extensions/agent-channel/poller.ts` → moves into `FileTransport`
- `pi/extensions/agent-channel/index.ts` → use `transport` + `display`

## Notes
- `createBackend()` becomes `createTransport()` + `createDisplay()`
- TmuxBackend.setup()/teardown() moves to TmuxDisplay
- File I/O helpers stay in transports.ts (used by FileTransport)
- `subscribe`/`unsubscribe` in the interface from day one — FileTransport implements
  via poller, push transports implement natively. Settles the design before 0015.2+.
