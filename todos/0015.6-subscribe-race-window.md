## Context

`UdsTransport.subscribe()` sends the subscribe frame asynchronously via
`ensureConnected().then(() => send(...))`. There's a brief window (milliseconds)
between `subscribe()` returning and the relay registering the subscription.
Messages published during this window are missed by push delivery.

The `catch_up=true` flag on `channel_watch` covers the gap for messages that
existed before subscribing, but not for messages published in the race window.

Observed in live testing: HTTP-published message arrived late because the
curl happened before the subscribe frame reached the relay.

## Options
- **Document only** — note the window in README/interfaces, acceptable for local dev
- **Make subscribe awaitable** — change `subscribe()` to return `Promise<void>`,
  `channel_watch` awaits it before returning. Breaking interface change.
- **Server-side buffering** — relay buffers messages for newly-subscribed clients
  for a short window. More complex.

## Notes
- In practice the window is ~1-5ms on local UDS
- catch_up covers pre-existing messages
- Only affects messages published during the exact subscribe registration window
