## Context

Repository research has different needs from docs crawling. For GitHub and local repos, the tool should prioritize README, docs, changelogs, examples, manifests, and code-map style summaries before deep file reads, while still preserving links between files, directories, and referenced resources.

## Acceptance Criteria

- [ ] Define the repository-research workflow for GitHub URLs and local repo paths, including what gets inspected first and how scope is narrowed.
- [ ] Specify an explicit first-pass priority order, such as `README*` → `docs/**` → `CHANGELOG*` → manifests/config → examples → targeted `src/**` reads.
- [ ] Specify how the tool builds a repo knowledge map from README files, docs directories, manifests, entrypoints, examples, and selected source files.
- [ ] Describe how cross-references are followed, such as README links, docs links, import relationships, release notes, and issue or PR references when needed.
- [ ] Clarify how this tool complements rather than duplicates existing `read`, `grep`, `find`, and `bash` workflows for local repos.
- [ ] Clarify the output model for repo summaries, file-level citations, and targeted deep reads so the agent can answer architecture and usage questions efficiently.
- [ ] Define safety and scope rules for binary files, large generated files, vendored directories, and other low-signal repository content.

## Affected Files

- `pi/extensions/...` — likely repo-ingestion and summarization modules
- `pi/README.md` — document GitHub/local repo research support if implemented
- `test/...` — future fixture-based repo research coverage

## E2E Spec

GIVEN a GitHub repo or local project path and a targeted architecture or usage question
WHEN the agent uses the research tool
THEN it can build a focused repo map, follow relevant references, and return source-cited answers without dumping the entire codebase

## Notes

- Prefer summary-first exploration over full source ingestion.
- Local repos should integrate cleanly with existing read/search tools rather than duplicate everything.
- The MVP should avoid pretending to be a universal code indexer; it should guide targeted reads.
