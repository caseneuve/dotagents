## Context

Manual multi-agent setup is possible but tedious. This task adds a convenience
path where one agent (or user command) can spawn another interactive Pi process
in the available mux backend, preconfigured to join the current agent-channel
communication context and follow a role prompt.

The original version of this todo targeted the superseded SQLite `agent-comm`
path. The migrated version targets the current `agent-channel` / relay
architecture and should support `*mux` backends rather than assuming tmux only:
`tmux` on Linux/portable shells and `cmux` where that is the natural local UI
(for example macOS setups).

## Acceptance Criteria

- [ ] There is a command or tool that spawns a new interactive Pi process in an
      available mux backend (`tmux` and/or `cmux`) with channel/lobby + role
      boot instructions.
- [ ] Backend selection is explicit and inspectable: auto-detect when possible,
      allow override, and report whether the child lives in tmux, cmux, or a
      supported fallback.
- [ ] Spawned agent startup is deterministic enough for repeated use: cwd,
      agent name or generated name, lobby/channel metadata, active transport
      config, and prompt directive are all controlled.
- [ ] The main frame reports where the spawned process lives (mux target/session,
      window/pane, or cmux equivalent) so the human can inspect it directly.
- [ ] Spawn failures are surfaced clearly and do not leave ambiguous state.
- [ ] E2E coverage verifies a spawned agent can send/receive over the same
      current `agent-channel` communication channel.

## Affected Files

- `pi/extensions/agent-channel/` — spawn command/tool integration with current
  channel/transport state.
- `shared/` or `scripts/` — shared mux helper if orchestration should be reused
  outside Pi.
- `pi/README.md` — spawn flow documentation for tmux/cmux behavior.
- `test/e2e/` — spawn + communication checks.

## E2E Spec

GIVEN an orchestrator agent in Pi with agent comms enabled
WHEN it spawns a reviewer/worker agent in mux communication mode
THEN the spawned agent appears in a known mux target and can exchange
agent-channel messages with the orchestrator

GIVEN both tmux and cmux are unavailable or spawn fails
WHEN the spawn command/tool is invoked
THEN the failure clearly explains what backend was attempted and how to recover

## Notes

Treat the mux backend as an inspectability and convenience layer, not the
message backend. The message backend remains `agent-channel` transport
selection: UDS/HTTP relay when available, FileTransport fallback otherwise.
