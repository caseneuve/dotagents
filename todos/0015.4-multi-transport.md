## Context

Agents may need to communicate with both local (same machine) and remote
(cross-machine) peers simultaneously. `MultiTransport` fans out publish to
multiple transports and merges reads. Transport selection auto-detects the
best available transport based on environment.

## Acceptance Criteria
- [ ] `MultiTransport` wraps multiple `MessageTransport` instances
- [ ] `publish` fans out to all transports
- [ ] `read` merges and deduplicates across transports
- [ ] `ack` dispatches to the transport that owns the message
- [ ] `subscribe` registers callback on all transports
- [ ] `createTransport()` factory with auto-detection logic and health probes
- [ ] Health probes: connect+disconnect for UDS (~500ms), `GET /channels` for HTTP (~500ms)
- [ ] Graceful fallback chain: UDS → HTTP → File (documented)
- [ ] Env vars: `AGENT_RELAY_URL`, `AGENT_UDS_SOCKET`
- [ ] Graceful degradation: if UDS/HTTP unavailable, fall back to file

## Affected Files
- `pi/extensions/agent-channel/transports.ts` — add MultiTransport, createTransport()
- `pi/extensions/agent-channel/transports.test.ts` — tests

## Notes
- Message dedup across transports: use message `id` field
- Transport that delivered the message should handle the ack
- Cross-transport message ordering is best-effort. For catch-up reads, sort by
  timestamp after dedup. Clock skew between machines is inherent — relay could
  assign server-side timestamps in a future version.
