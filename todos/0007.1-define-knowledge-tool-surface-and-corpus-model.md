## Context

Before implementation, we need a crisp tool contract for how the agent fetches, searches, reads, and cites external knowledge. The design should prefer structured, composable tools over a browser-like interface and define how fetched material becomes a reusable corpus instead of one-off page dumps.

## Acceptance Criteria

- [ ] Define the proposed tool surface area, such as `knowledge_discover`, `knowledge_collect`, `knowledge_search`, and `knowledge_read`, or an explicitly justified alternative.
- [ ] Specify request/response shapes for each tool, including task input, filters, source metadata, chunk IDs, and error modes.
- [ ] Include at least one concrete JSON schema example for each tool request and response shape.
- [ ] Define the corpus/chunk model, including heading-aware chunking, code-block preservation, dedupe rules, and source citation fields.
- [ ] Make the citation contract explicit and mandatory, including source URL or path, heading path, chunk ID, and timestamp or version hash for every returned chunk.
- [ ] Clarify how fetched content is cached and reused across follow-up research within a session or project.

## Affected Files

- `pi/extensions/...` — likely module boundary for the future extension design
- `pi/README.md` — document the tool surface if implementation follows
- `test/...` — future schema and behavior coverage

## E2E Spec

GIVEN a focused docs-research task
WHEN the agent invokes the knowledge toolset
THEN each tool has a clear contract for discovering, collecting, searching, and reading source-cited content from a reusable corpus

## Notes

- Prefer heading-aware chunks over arbitrary token windows.
- Source traceability should include URL or path, section path, stable chunk ID, and freshness metadata.
- The design should be strict enough that downstream tools can rely on citation fields always being present.
