## Context

The extracted packages must be proven independently before dotagents removes its broad extension,
theme, and prompt paths. Validation needs to cover clean local installs, pinned Git installs,
resource filtering, reload behavior, dependencies, and collision-free composition.

## Acceptance Criteria

- [ ] Test every package with an isolated Pi configuration directory and a clean dependency state.
- [ ] Confirm `pi list` and `pi config` show expected packages and independently filterable resources.
- [ ] Verify local-path installation for development and pinned Git installation for reproducible use.
- [ ] Verify package commands, tools, shortcuts, skills, prompts, and themes against the pre-extraction behavior baseline.
- [ ] Verify `/reload`, session shutdown, and fresh startup produce no package/resource diagnostics.
- [ ] Compose the intended global defaults without duplicate commands, tools, footer owners, or editor owners.
- [ ] Document any deliberate behavior changes and the package-set versions/refs approved for cutover.

## Affected Files

- New repositories under `~/git/pi/` — package test targets.
- Package tests and READMEs — validation evidence and installation instructions.
- `pi/README.md` — approved package set reference if needed.

## Notes

Do not load an extracted extension and its original dotagents copy in the same normal validation
session. The actual settings migration belongs to todo `0046`.
