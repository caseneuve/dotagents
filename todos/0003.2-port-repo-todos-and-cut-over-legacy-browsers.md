## Context

Once the shared browser helpers and `agent-journal-next` shape are proven,
port `repo-todos` onto the same modular structure. After both next-generation
extensions reach parity, swap the canonical command paths over and remove the
old single-file implementations so the repo keeps one obvious implementation per
browser.

## Acceptance Criteria

- [ ] `pi/extensions/repo-todos-next/` reuses the shared overlay/layout/filter
      helpers instead of re-copying logic from the old monolith.
- [ ] `repo-todos-next` has a documented parity checklist against the current
      browser, including tree folding, sort mode, hide/show completed,
      preview focus behavior, preview paging, filtering, path rendering, and
      frame/border behavior that materially affects UX.
- [ ] The cutover happens only after both next-generation browsers are feature-
      parity checked and docs point to the new canonical paths.
- [ ] Cutover updates bootstrap/discovery/docs without leaving duplicate
      command implementations discoverable for the same browser.
- [ ] The cutover leaves one canonical implementation for each browser and the
      legacy monolith files are removed.

## Affected Files

- `pi/extensions/repo-todos-next/` — modularized replacement
- `pi/extensions/shared/` — reused and possibly extended helpers
- `pi/extensions/repo-todos.ts` — reference behavior before removal
- `pi/extensions/agent-journal.ts` — may be removed once both cutovers finish
- `pi/README.md` — update docs after the canonical swap

## Notes

Do not cut over early. Use the existing working extensions as the behavioral
baseline until both next-generation implementations feel stable enough to
replace them.
