## Context

Once the design is broken into tool surface, docs crawling, and repo research, we still need a concrete implementation path for this repo. That includes extension structure, dependencies, cache layout, and a test plan that fits the repo's current FCIS and fixture-heavy testing style.

## Acceptance Criteria

- [ ] Identify the MVP scope that delivers useful research value without requiring a full browser or search engine stack.
- [ ] Propose the likely extension/module layout under `pi/extensions/` and any shared helpers that should live outside a single extension file.
- [ ] Define the cache/corpus storage location, invalidation strategy, and project-vs-global behavior.
- [ ] Specify the required pre-implementation fixture set for docs crawling, link ranking, and repo-research flows.
- [ ] Outline unit and E2E test coverage using repo-appropriate fixtures and tasks, including what should be tested before implementation begins.
- [ ] Define pass criteria for the fixture set, such as expected retrieval hits, citation completeness, and acceptable crawl noise.

## Affected Files

- `pi/extensions/...` — future extension entrypoint and helpers
- `pi/README.md` — implementation-facing docs once built
- `test/unit/...` — pure helper tests for chunking, ranking, and corpus logic
- `test/e2e/...` — fixture-based end-to-end coverage for docs/repo research flows

## Notes

- Keep host writes out of E2E tests.
- Prefer pure logic in testable helper modules and push I/O to the boundary.
- This task should consolidate the design work from `0007.1`, `0007.2`, and `0007.3` into an implementation-ready plan.
- Favor deterministic fixtures over live-network assertions wherever possible.
