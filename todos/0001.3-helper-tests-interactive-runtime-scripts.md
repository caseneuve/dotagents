## Context

The remaining helpers depend on subprocesses, stdin/stdout JSON contracts, editor/runtime context, or tmux state. They still need coverage, but the tests will need tighter fixture control:

- `shared/skills/code-review/detect-and-lint.sh`
- `shared/skills/org-journal/new-entry.bb`
- `shared/skills/pk-tmux/tmux-create.sh`
- `shared/skills/pk-tmux/tmux-run.sh`
- `shared/skills/pk-tmux/tmux-status.sh`
- `shared/skills/pk-tmux/tmux-wait.sh`
- `claude/hooks/smart-lint.sh`
- `claude/statusline-command.sh`

## Acceptance Criteria

- [ ] `new-entry.bb` gets direct coverage via `bb test`, either as-is or after extracting pure helpers into a require-able namespace.
- [ ] The tmux helpers have end2edn coverage for session creation, command execution, output capture, and missing-session/window errors.
- [ ] `detect-and-lint.sh`, `smart-lint.sh`, and `statusline-command.sh` have fixture-driven tests for their key input/output contracts.
- [ ] Any required test doubles for `git`, `jj`, `tmux`, `jq`, or lint tools are kept minimal and readable.

## Affected Files

- `shared/skills/code-review/detect-and-lint.sh` - project detection and test-safety output
- `shared/skills/org-journal/new-entry.bb` - pure/project-context gathering logic
- `shared/skills/pk-tmux/*.sh` - tmux orchestration and output capture
- `claude/hooks/smart-lint.sh` - hook JSON blocking/warning behavior
- `claude/statusline-command.sh` - statusline rendering from JSON input
- `test/unit/...` - Babashka unit tests for extracted logic
- `test/e2e/cases.edn` - command-level CLI scenarios
- `test/fixtures/...` - stdin payloads, fake tool outputs, and repo fixtures

## Notes

For shell helpers with complex branching, consider extracting only the pure classifier/formatter pieces into Babashka and leave the shell entrypoint thin. That keeps runtime behavior intact while making tests cheaper.
