## Context

The first workflow that should justify the framework is a review loop: a worker
makes or updates code, a reviewer inspects it, and the worker iterates until
review passes or a round limit is reached. For this to be reliable, the
handoff between worker and reviewer cannot rely only on vague freeform prose;
it needs a structured contract that the framework can parse and use to drive the
loop.

This task is the orchestration layer that turns the spec/session/trace work into
a genuinely useful workflow.

## Acceptance Criteria

- [ ] The framework provides a canonical `review_loop` workflow that runs a
      worker and reviewer as isolated SDK-backed subagent sessions.
- [ ] The workflow enforces a structured reviewer verdict contract with at
      least pass/fail plus machine-readable findings or follow-up instructions.
- [ ] The workflow enforces a structured worker handoff contract with at least a
      status, changed-files summary, and notes needed for the next review round.
- [ ] The loop stops deterministically on reviewer pass, worker/reviewer error,
      abort, or configured max rounds, and the stopping reason is recorded in
      the workflow trace.
- [ ] The parent-facing result returned to the main agent is a compact summary
      of the loop outcome rather than the full nested conversation.
- [ ] Tests cover at least pass-on-first-review, fail-then-fix-then-pass, and
      max-rounds-reached behavior.

## Affected Files

- `pi/extensions/subagents/review-loop.ts` — workflow orchestration
- `pi/extensions/subagents/session-factory.ts` — SDK-backed subagent session
  creation
- `pi/extensions/subagents/protocols.ts` — reviewer/worker structured handoff
  parsing and formatting
- `pi/extensions/subagents/index.ts` — parent-facing tool registration
- `.pi/agents/reviewer.md` and `.pi/agents/worker.md` — canonical sample agent
  contracts, with user-level agent locations treated only as compatibility
  inputs to discovery
- `test/unit/pi/subagents/` — protocol parsing and loop-planning tests
- `test/e2e/pi/` — end-to-end review loop scenarios

## E2E Spec

GIVEN Markdown-defined worker and reviewer agents that follow the framework
contracts
WHEN the parent agent runs `review_loop` for a coding task
THEN the worker and reviewer iterate until the reviewer passes or the workflow
hits its configured stop condition

GIVEN a review loop that produces findings and then fixes them
WHEN the workflow completes
THEN the parent agent receives a compact summary of the final verdict, rounds
used, and changed files, while the human can inspect the detailed round trace
separately

## Notes

Favor turn-based orchestration over trying to emulate a freeform simultaneous
chat between agents. The framework should mediate clear round handoffs and keep
loop conditions machine-checkable.
