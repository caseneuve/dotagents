## Context

The biggest gap versus Playwright is not just clean extraction, but the ability to see useful links on a page and continue exploring efficiently. We need a docs-oriented discovery model that identifies meaningful next links, avoids boilerplate navigation noise, and follows a task-directed crawl policy.

## Acceptance Criteria

- [ ] Define how the tool extracts visible and meaningful links from fetched docs pages, including link text, URL, context heading, and inferred link kind.
- [ ] Specify at least 3-5 deterministic ranking heuristics for likely useful next hops, such as changelog pages, migration guides, API docs, examples, and in-content references.
- [ ] Define tie-break rules when multiple candidate links score similarly.
- [ ] Define crawl controls including allow/deny scope, host boundaries, depth, max pages, revisit policy, and loop/dedup protection.
- [ ] Document operational constraints including timeout, retry, rate-limit behavior, robots or site-policy stance, and cache TTL or refresh rules.
- [ ] Document how the tool distinguishes content links from low-value navigation, footer, auth, social, or marketing links.
- [ ] Define measurable evaluation targets for discovery quality, such as precision@k, retrieval hit rate on fixtures, or a maximum crawl-noise ratio.

## Affected Files

- `pi/extensions/...` — likely docs crawling and extraction modules
- `pi/README.md` — explain discovery/follow behavior if implemented
- `test/...` — future crawl ranking and dedupe coverage

## E2E Spec

GIVEN a docs landing page and a research goal such as breaking changes for extensions
WHEN the agent runs discovery and collection
THEN the tool can identify relevant links, follow the best candidates, and build a focused corpus without browser-style manual navigation

## Notes

- The design should support both single-page discovery and recursive collection.
- Link extraction should preserve enough context to explain why a page was followed.
- The ranking model should stay simple and explainable in MVP rather than relying on opaque scoring.
- Safety constraints should cover external-host boundaries and obvious prompt-injection or hostile-content handling in fetched pages.
