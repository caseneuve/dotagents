## Context

Central relay that agents connect to for push-based message delivery. Runs on
Raspberry Pi (or any machine). Listens on both UDS (local) and HTTP (remote).
Channels are the only addressing primitive — server is a dumb keyed pub/sub broker.

## Acceptance Criteria
- [ ] Server listens on UDS socket (configurable path, default `/tmp/agent-channels.sock`)
- [ ] Server listens on HTTP port (configurable, default 7700, `--bind` flag for interface)
- [ ] `POST /channels/:channel/messages` — publish, fans out to all subscribers
- [ ] `PATCH /channels/:channel/messages/:id` — ack
- [ ] `GET /channels/:channel/stream` — SSE subscription, pushes new messages live
- [ ] `GET /channels` — list all channels with message counts
- [ ] UDS protocol: NDJSON (newline-delimited JSON, one object per `\n`)
- [ ] Messages stored in memory with configurable max per channel (default 1000)
- [ ] Server is a single Bun/TypeScript entrypoint
- [ ] No-auth threat model documented: "only run on trusted networks"

## Affected Files
- `shared/relay/server.ts` — new, relay server entrypoint
- `shared/relay/store.ts` — new, in-memory channel store
- `shared/relay/server.test.ts` — new, tests

## Notes
- Bun has native UDS support via `Bun.listen` with `unix` option
- Bun has native HTTP server — no framework needed
- SSE is just `Transfer-Encoding: chunked` with `text/event-stream` content type
- UDS framing: NDJSON — simple, debuggable with `socat`, no custom parser
- In-memory store is sufficient for v1 — relay restart loses undelivered messages.
  `Last-Event-ID` SSE reconnection helps for transient disconnects but not restarts.
  Lightweight WAL/append-only file is a v1.1 follow-up if needed.
- No auth: acceptable for trusted home LAN. Document explicitly. `--bind` flag
  allows restricting to specific interface (e.g. `192.168.x.x` instead of `0.0.0.0`).
- Lives in `shared/relay/` not `relay/` — serves all runtimes, not just pi.
