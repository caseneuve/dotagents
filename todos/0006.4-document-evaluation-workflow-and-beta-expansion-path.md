## Context

Even a good parser spike is only worth keeping if it improves real work. We
need lightweight documentation that explains how to evaluate the Python-first
POC in actual agent sessions, what success looks like, and how the POC would
expand into a broader beta if the experiment is promising.

This task turns the implementation into an intentional experiment rather than a
one-off local hack.

## Acceptance Criteria

- [ ] The repo documents how to load or enable the Tree-sitter POC extension in
      Pi and which Python-focused tools it provides.
- [ ] The docs define concrete evaluation criteria such as faster file
      understanding, better edit targeting, and less unnecessary full-file
      reading.
- [ ] The docs include at least one concrete evaluation checklist or demo
      workflow that someone can run to judge whether the POC is actually useful
      in a real agent session.
- [ ] The docs capture known limits of the POC, including syntax-only
      understanding and the lack of semantic/LSP guarantees.
- [ ] The beta path is explicit about what comes next if the POC is successful,
      including adapter generalization and criteria for adding more languages.
- [ ] The documented path preserves the repo's preference for small focused Pi
      extensions over a monolithic all-in-one system.

## Affected Files

- `pi/README.md` — primary user-facing documentation for the POC and evaluation
- `README.md` — mention only if the top-level repo docs should point to the POC
- `pi/extensions/treesitter/README.md` — optional extension-local notes if they
  stay narrowly implementation-focused

## Notes

Document both the yes-case and the no-case: if the POC does not materially help
agent workflows, the repo should say so and avoid carrying a misleading
half-supported feature forward.
