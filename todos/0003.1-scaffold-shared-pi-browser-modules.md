## Context

Start by carving out the reusable pieces from the journal/todo overlay work:
text helpers, frame rendering helpers, split-layout calculations, and shared
filter-input behavior. Use those pieces to build a modularized
`agent-journal-next` extension while keeping the current `agent-journal.ts`
working as the reference implementation.

## Acceptance Criteria

- [ ] `pi/extensions/shared/` exists with small reusable helpers for browser
      overlays (text/layout/frame/filter behavior), with pure logic separated
      from side effects where practical.
- [ ] Shared modules are extracted only for behavior duplicated across both
      browsers; avoid generic abstractions for behavior that exists in only one
      extension.
- [ ] `pi/extensions/agent-journal-next/` is directory-based and wired through
      `index.ts` plus focused modules such as scanning/parsing/component code.
- [ ] `agent-journal-next` has a documented parity checklist against the
      current `agent-journal`, including project filtering, preview paging,
      list/preview focus behavior, and editor handoff.
- [ ] Extracted shared helpers are tested directly when they are pure.

## Affected Files

- `pi/extensions/shared/` — extracted helpers
- `pi/extensions/agent-journal-next/` — modularized replacement
- `pi/extensions/agent-journal.ts` — reference behavior to compare against
- `pi/README.md` — temporary docs for the transition if needed

## Notes

Keep the abstraction lightweight. Prefer shared helpers and pure functions over
building a large generic browser framework too early.
