## Context

Before tools and automation, we need one clear message protocol and storage
contract. This task defines message fields, channel semantics, and the SQLite
schema/query patterns that all communication tools share.

## Acceptance Criteria

- [ ] A normalized message schema is defined (id, channel, from, to,
      type, body, correlation/thread id, timestamps, ack state).
- [ ] SQLite storage and read/write semantics are implemented for single-host,
      multi-process agent communication.
- [ ] Basic lifecycle helpers exist: create channel metadata (if needed), append
      message, fetch pending messages, ack message, and optional cleanup.
- [ ] Concurrency behavior is documented (busy timeout/retry strategy,
      ordering guarantees, duplicate prevention where applicable).
- [ ] Unit tests cover serialization, persistence, ordering, and ack handling.

## Affected Files

- `pi/extensions/agent-comm/` — protocol types and SQLite storage layer
- `test/unit/pi/agent-comm/` — protocol and storage tests
- `pi/README.md` — protocol and storage documentation

## E2E Spec

GIVEN two processes writing to and reading from the same SQLite communication
store
WHEN messages are sent and acknowledged on a channel
THEN each process can retrieve intended messages in deterministic order without
losing acknowledged state

Closed as superseded. The SQLite store/protocol path was replaced by the current agent-channel message protocol plus relay-backed transports. Future comms work should extend `pi/extensions/agent-channel/` and `shared/relay/`, not create `pi/extensions/agent-comm/`.
