## Context

Before building the parser spike, we should decide what kind of extension we
are actually trying to build. The POC should be Python-first, but not Python-
locked. That means defining a small generic core that handles parser selection,
file loading, output normalization, and tool registration, plus a language-
specific adapter layer that can start with Python and grow later.

This task is about pinning down the architecture and stable tool contracts so
later work does not hard-code Python assumptions into every module.

## Acceptance Criteria

- [ ] The repo has one named canonical design artifact for this POC, such as
      `pi/extensions/treesitter/DESIGN.md`, and it describes the module
      boundaries and responsibilities clearly enough to guide later work.
- [ ] The POC has a documented module boundary between generic extension logic
      and Python-specific query/extraction logic.
- [ ] We choose and document the primary implementation approach for the spike:
      TypeScript/Node Tree-sitter integration first, with any fallback helper
      process approach recorded explicitly as a contingency rather than an
      implicit design.
- [ ] The first tool contracts are defined for at least `outline`, `context`,
      and `find`, including compact output schemas suitable for agent use.
- [ ] The design explicitly states that v0 is read-only and inspection-focused,
      not a semantic refactoring system.
- [ ] The architecture notes identify where future adapters for TS/JS, Rust,
      or other languages would plug in without changing the public tool shape.

## Affected Files

- `pi/extensions/treesitter/` — extension entrypoint, adapter interfaces, and
  design-adjacent module layout
- `pi/extensions/treesitter/DESIGN.md` — canonical POC architecture and tool
  contract document
- `pi/README.md` — canonical project docs if architecture decisions are made
  user-visible
- `test/unit/pi/treesitter/` — tests for any pure schema/normalization helpers

## Notes

Keep the initial tool surface small. A good v0 contract is more valuable than a
large unstable one.
