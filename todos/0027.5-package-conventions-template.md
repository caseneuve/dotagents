## Context

The extension extraction epic will create several independently installable repositories under
`~/git/pi/`. They need one consistent package layout that works first with local-path and Git
installs and can support npm publication later without redesign.

The agreed repository model is one repository per logical package cluster. Experimental work is
a lifecycle state: scaffold experiments remain outside the explicit Pi manifest and are run only
with `pi -e`; installable experiments use `0.x`, clear status documentation, pinned Git refs, and
are never part of global bootstrap defaults.

## Acceptance Criteria

- [ ] Define a reusable repository template with an explicit `pi` manifest and no accidental resource auto-discovery.
- [ ] Define dependency policy: Pi APIs in `peerDependencies` with `"*"`, runtime libraries in `dependencies`, and development-only tooling in `devDependencies`.
- [ ] Define shared TypeScript, formatting, test, README, license, and Linux/macOS support conventions.
- [ ] Document local-path installation as the development default and pinned Git installation as the initial reproducible distribution path.
- [ ] Document the experimental lifecycle: scaffold, experimental `0.x`, stable personal, and optionally published.
- [ ] Ensure experimental resources are excluded from stable manifests unless explicitly promoted.
- [ ] Record how future npm publication can be added without changing package boundaries.

## Affected Files

- New package repositories under `~/git/pi/` — template consumers.
- Package README and manifest scaffolding — canonical package conventions.
- `pi/README.md` — link to the package conventions if dotagents needs to describe them.

## Notes

Do not create a monorepo for the initial local/Git-first distribution. Pi installs Git repositories
at the repository root, so one repository per installable package is the cleanest model.
